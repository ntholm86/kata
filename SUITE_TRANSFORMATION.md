# Old Suite vs New Suite

## The Short Answer

The suite changed a lot in structure.
It changed less in intent.

The old suite was a broader TPS-style improvement toolkit.
The new suite is a smaller, principle-derived system for delegable autonomy.

## Side-by-Side

| Aspect | Old Suite (v1) | New Suite (v2) | What it means |
|--------|-----------------|----------------|---------------|
| **Shape** | 8 live skills | 5 live skills | The suite was compressed to the parts that embody the core principles directly |
| **Live skills** | `kata`, `kaizen`, `kaikaku`, `hansei`, `shiken`, `mura`, `muri`, `muda`, plus `project-increment` in the live structure | `kata`, `kaizen`, `kaikaku`, `hansei`, `shiken` | The 3M lenses and project increment no longer live as separate operational workflows |
| **Diagnostic model** | Mura, Muri, and Muda often ran like a fixed procedure | Unevenness, overburden, and waste live inside the reasoning vocabulary | The agent is expected to use the lenses when useful, not perform them ritualistically |
| **Kata's role** | Thicker orchestrator with stronger procedural feel | Thinner orchestrator focused on grasp, diagnose, decide, execute, record, persist | Kata became more mission-oriented and less like a prescribed treatment script |
| **Kaizen's role** | Heavier audit/rating/improvement loop with more rubric machinery | Simpler improvement skill: diagnose, challenge blind spots, prioritize, implement, verify | Kaizen became easier to carry in the head and closer to the core job of incremental improvement |
| **Shiken's role** | Important, but more peripheral to day-to-day improvement | Central to proving the system is reasoning rather than complying | Once the problem was defined clearly, ARF validation became core rather than optional |
| **Trail model** | Strong GENBA focus, but the full multi-resolution trail was less explicit | Kiroku + decision index + summary + GENBA as a layered evidence architecture | Observability became more explicit and more faithful to Principle 2 |
| **Measurement** | Strong emphasis on internal improvement and rubric evolution | Stronger split between input quality and output quality | The suite now distinguishes “good skills” from “good work done under those skills” |
| **Primary identity** | Improvement toolkit | Adoption architecture / delegable autonomy framework | The suite now reads more clearly as a system for trust, evidence, and bounded authority |

## What Stayed The Same

These things did **not** fundamentally change:

- the desire for disciplined improvement
- the need for visible reasoning
- the need for reflection when the loop plateaus or lies to itself
- the idea that the work must be provable, not just persuasive
- the belief that human trust is earned through evidence, not demanded through confidence

## What Changed Most Deeply

The biggest change was not the skill count.

The biggest change was the suite's self-understanding.

Before, it behaved mainly like a system for improving artifacts.

Now, it behaves like a system for making autonomous work **delegable**.

That means:

- mission matters more than checklist
- trail matters more than summary alone
- external validation matters more than self-score
- authority must stay bounded and revocable

## Why The Simplification Matters

The simplification was not cosmetic.

It removed a major failure mode: turning diagnostic vocabulary into ritual procedure.

The old suite risked saying:
"run these lenses in this order because that is the process."

The new suite says:
"understand the mission, use the vocabulary that helps, show your reasoning, and prove the result."

That is much closer to Commander's Intent.

## Best One-Line Summary

The suite used to look like a toolbox for improving things.

Now it looks like an operating system for earned, explainable, delegable AI authority.

## So Did It Change Dramatically?

Yes in form.

No in essence.

It became more coherent, more principle-shaped, and more honest about what it was trying to build all along.
